# Требования по разработке фронтенда

## Файловая структура

Код организован в два уровня: **уровень страницы** и **общепроектный уровень**.

### Уровень страницы

Каждая страница — это директория в `src/pages/`. Страницы группируются по типу лэйаута:

- `src/pages/public/` — страницы для неавторизованного пользователя (публичный лэйаут)
- `src/pages/protected/` — страницы для авторизованного пользователя (внутренний лэйаут)

Лэйаут группы живёт в поддиректории `_layout/` внутри группы:

```
src/pages/
├── public/
│   ├── _layout/
│   │   └── PublicLayout.tsx
│   ├── LoginPage/
│   └── RegisterPage/
└── protected/
    ├── _layout/
    │   └── ProtectedLayout.tsx
    ├── TasksPage/
    └── ...
```

Структура страницы:

```
src/pages/protected/TasksPage/
├── index.ts              # реэкспорт: export { TasksPage } from './TasksPage'
├── TasksPage.tsx         # основной компонент страницы
├── components/           # компоненты, специфичные только для этой страницы
├── hooks/                # хуки, специфичные только для этой страницы
└── utils/                # утилиты, специфичные только для этой страницы
```

Поддиректории (`components/`, `hooks/`, `utils/`) создаются по мере необходимости — не заранее.

### Общепроектный уровень

Всё, что переиспользуется на нескольких страницах, живёт в `src/lib/`:

```
src/lib/
├── components/           # переиспользуемые UI-компоненты
├── hooks/                # переиспользуемые хуки
└── utils/
    ├── date/             # утилиты для работы с датами
    ├── format/           # форматирование (числа, валюта, строки)
    └── ...               # другие категории по смыслу
```

Утилиты разделены по категориям-поддиректориям, чтобы избежать нагромождения.

### Соглашения по именованию файлов

Правила распространяются на компоненты, хуки и утилиты:

- **1 сущность = 1 файл**: каждый компонент, хук или утилита живёт в отдельном файле.
- **Имя файла совпадает с именем сущности** (без учёта регистра): `TaskCard.tsx` для `TaskCard`, `useTaskFilter.ts` для `useTaskFilter`, `formatCurrency.ts` для `formatCurrency`.

### Экспорты и объявление компонентов

- **Только именованные экспорты** — `export default` не используется нигде в проекте.
- **Все компоненты и страницы — стрелочные функции**, типизированные как `React.FC`:
  ```tsx
  export const TasksPage: React.FC = () => {
    return <div />;
  };
  ```

### Правило перемещения

Код пишется на уровне страницы. Как только он понадобился на второй странице — переносится в `src/lib/`.

## Навигация

- Хардкодные строки роутов запрещены. Все пути определяются как константные объекты-перечисления с функциями:
  - **Страницы** — `src/lib/AppRoute.ts`:
    ```ts
    export const AppRoute = {
      Tasks: () => '/tasks',
      Login: () => '/auth/login',
    } as const;
    ```
  - **API-эндпоинты** — `src/lib/ApiRoute.ts`:
    ```ts
    export const ApiRoute = {
      Login: () => '/api/auth/login',
      Tasks: () => '/api/tasks',
    } as const;
    ```
- Везде в коде используется `AppRoute.RouteName()` / `ApiRoute.RouteName()` вместо строковых литералов.

## Написание стилей

### clsx и cn

- Если набор классов содержит менее 2 групп, или ни одна группа не содержит 3+ свойств — используем обычную строку без обёртки:
  ```tsx
  <div className="flex items-center gap-2" />
  ```
- Если набор классов содержит 2 и более групп, и хотя бы одна группа содержит 3+ свойств — оборачиваем в `clsx()` и разбиваем на группы по смыслу, каждая группа на отдельной строке:
  ```tsx
  <div
    className={clsx(
      'absolute top-0 left-0 z-10',
      'flex items-center gap-2',
      'w-full px-4 py-2',
      'text-sm font-medium',
      'bg-neutral-800 text-neutral-100',
    )}
  />
  ```
- `cn()` (clsx + tailwind-merge) используем только когда компонент принимает внешние классы, которые могут переопределять внутренние:
  ```tsx
  <div className={cn('bg-neutral-800 px-4 py-2', className)} />
  ```

### Порядок групп классов

1. Позиционирование (`position`, `top/right/bottom/left`, `z-index`)
2. Дисплей и раскладка (`display`, `flex`, `grid`, `gap`, `align`, `justify`)
3. Блочная модель (`width`, `height`, `margin`, `padding`, `border`, `overflow`)
4. Визуальные свойства (`background`, `color`, `opacity`, `shadow`, `ring`)
5. Типографика (`font`, `text`, `leading`, `tracking`, `whitespace`)
6. Анимации и переходы (`transition`, `animate`, `duration`, `ease`)
7. Прочее

### Произвольные значения

Arbitrary-значения (`[1600px]`, `[24rem]`) не используются, если существует стандартный класс Tailwind. Например, `max-w-400` вместо `max-w-[1600px]`, `gap-2` вместо `gap-[8px]`. Произвольные значения допустимы только когда значение не попадает в стандартную шкалу.

### Сортировка

Используем `prettier-plugin-tailwindcss` — он автоматически сортирует классы внутри каждой строки в соответствии с порядком выше.

## Автотесты

- Тестами покрываем только чистые функции: вычисления, форматирование, преобразования данных и т.д.
- Компоненты, хуки и роутинг тестами не покрываются.
